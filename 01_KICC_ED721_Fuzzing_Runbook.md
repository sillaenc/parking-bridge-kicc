# KICC ED-721 RS-232 퍼징 실행 가이드(직원용)

## 개요(우리가 왜 이걸 하고 있는가)
- ED-721 단말기와 POS(라즈베리파이/노트북) 간 RS-232 연동을 위해 **신전문 패킷 포맷**을 맞춰야 합니다.
- 현재는 연결 자체는 되지만, 단말이 우리가 보낸 전문을 **NAK(0x15)** 로 거부하고 있어 “정확히 어떤 LEN/CRC 규칙을 요구하는지”가 확정되지 않았습니다.
- 그래서 이 작업은 단말이 수용하는 조합을 찾기 위한 **탐색 단계**이며, 최종 목표는 **ACK(0x06)** 가 나오는 “정답 조합”을 확보해 이후 `payment_kicc.py`(정식 모듈)로 옮기는 것입니다.

## 목적(요청 배경)
- **목표**: KICC ED-721 단말기에 대해, 우리가 추정한 신전문 포맷에서 **ACK(0x06)** 가 발생하는 **정확한 조합(LEN 계산/엔디안, CRC 바이트 순서, CRC 값)**을 찾는다.
- **현재 상황**: RS-232 연결은 되었으나, 단말이 우리가 만든 패킷을 **NAK(0x15)** 로 거부하는 상태다. 따라서 “맞는 조합”을 찾기 위해 전수 탐색(퍼징)을 수행한다.
- **성공 조건**: 콘솔에 **`[FOUND]`** 로그가 뜨고, 같은 폴더/지정 경로에 **`kicc_fuzz_found.json`** 파일이 생성되며 스크립트가 종료된다.

## 준비물 / 사전 조건
- **MacBook (또는 Linux) + Python 3**
- **USB-to-Serial 컨버터**(RS-232 또는 TTL 변환 타입은 현장 구성에 따름)
- **KICC ED-721 단말기**
- 전달받는 파일(2개):
  - **코드**: `kicc_crc_fuzzer.py`
  - **문서**: `01_KICC_ED721_Fuzzing_Runbook.md`

> 이 작업은 “폴더 전체/레포 전체 공유”를 전제로 하지 않습니다. 위 2개 파일만 있으면 실행 가능합니다.

## 하드웨어/배선 체크리스트(필수)
- **GND 연결**: 컨버터 GND ↔ 단말 GND
- **TX/RX 교차 연결**: 컨버터 TX ↔ 단말 RX / 컨버터 RX ↔ 단말 TX
- **단말 전원 안정성**: 전원 불안정하면 포트가 자주 끊기며 퍼징이 느려진다.
- **단말 모드 확인(가능한 경우)**: 단말 설정에서 RS-232/POS 연동 모드가 꺼져 있으면 ACK가 절대 안 나올 수 있다.

## 소프트웨어 체크리스트(필수)
- **동일 포트를 잡는 프로그램 종료**
  - 시리얼 모니터 앱, 다른 파이썬 스크립트, IDE 플러그인 등이 포트를 점유하면 오류가 난다.
- **포트 경로 확인**
  - 예: `/dev/tty.usbserial-B000EYUJ`
  - 모르면 아래 명령으로 확인:

```bash
ls -1 /dev/tty.* | grep -E "usb|serial" || true
```

## 실행(기본 권장 커맨드)
### 0) 작업 폴더 준비(예시)
두 파일(`kicc_crc_fuzzer.py`, `01_KICC_ED721_Fuzzing_Runbook.md`)을 임의의 폴더에 두고 그 폴더로 이동합니다.

```bash
cd /path/to/working_dir
```

### 1) 가상환경 준비(권장)
```bash
python3 -m venv venv
source venv/bin/activate
pip install -U pip pyserial
```

### 2) 퍼징 실행
아래 커맨드를 **그대로 복사/실행**합니다.

```bash
python3 kicc_crc_fuzzer.py \
  --port /dev/tty.usbserial-B000EYUJ \
  --resume \
  --resp-timeout-sec 0.12 \
  --delay-sec 0.005
```

### 파라미터 의미(변경 금지 권장)
- **`--resume`**: 중간에 끊겨도 이어서 재개(체크포인트 사용)
- **`--resp-timeout-sec 0.12`**: 응답을 기다리는 최대 시간(짧을수록 빠르지만 응답을 놓칠 수 있음)
- **`--delay-sec 0.005`**: 패킷 사이 간격(너무 작으면 컨버터/단말이 불안정해질 수 있음)

## 정상 동작 로그 예시 / 해석
- **정상 진행**
  - `"[MODE] index=..."`
  - `"[CONNECTED] serial port opened."`
  - `"[CKPT] packets=1024 ... crc=0x...."`

- **가끔 떠도 괜찮은 경고(자동 재연결됨)**
  - `"[WARN] serial error: device reports readiness to read but returned no data ... (retrying in 1s)"`
  - 의미: macOS에서 USB-시리얼이 순간적으로 끊기거나, 포트가 일시 오류를 낸 것. 스크립트가 **재연결 후 계속 진행**하면 OK.

- **성공(즉시 보고/중단 가능)**
  - `"[FOUND] ACK detected. Saved: kicc_fuzz_found.json"`
  - 이 로그가 뜨면 이미 결과 파일이 저장된 상태다.

## 중단/재개 방법
- **중단**: 터미널에서 `Ctrl + C`
- **재개**: 동일 커맨드로 다시 실행(반드시 `--resume` 포함)

---

## 산출물(반드시 전달해야 하는 파일)
실행 후 아래 파일들을 **그대로** 전달한다(내용 수정 금지).
- **체크포인트(진행 상황)**: `kicc_fuzz_checkpoint.json`
- **성공 결과(생기면 가장 중요)**: `kicc_fuzz_found.json`

추가로 아래 정보도 함께 전달:
- 사용한 포트 경로(예: `/dev/tty.usbserial-B000EYUJ`)
- 케이블/컨버터 종류(가능하면 사진 1장)
- 실행 시작/종료 시간
- 경고가 얼마나 자주 떴는지(대략적인 빈도)

## 문제 발생 시(직원이 해야 할 것)
### 1) `[WARN] ... multiple access on port?`가 반복적으로 뜰 때
- 다른 프로그램이 포트를 잡고 있는지 확인하고 모두 종료한다.
- USB 포트를 다른 곳으로 바꾸고 다시 시도한다.

### 2) 로그가 멈춘 것처럼 보일 때
- 최소한 **`[CKPT]`가 1024 패킷 단위로 계속 올라가는지** 확인한다.
- `[CKPT]`가 10분 이상 갱신되지 않으면 `Ctrl+C`로 중단 후 동일 커맨드로 재실행(`--resume`).

### 3) 몇 시간 돌려도 `kicc_fuzz_found.json`이 안 생길 때
- 그 상태 자체가 중요한 정보다.  
  체크포인트 파일과 함께 “현재 packets/모드 진행률”을 공유한다.

## 안전/주의 사항
- 장시간 퍼징은 **단말/컨버터에 부하**가 갈 수 있다. 발열이 심하면 잠시 중단하고 식힌다.
- 현장 결제/승인과 혼동되지 않도록, 테스트 환경에서만 수행한다.
- 단말이 실제 결제 모드로 들어갈 가능성이 있으면(표시/삑 소리 등), 즉시 중단하고 상황을 공유한다.
