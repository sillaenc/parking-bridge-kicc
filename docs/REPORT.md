## 1. 요약

KICC ED-721 결제 단말의 시리얼 통신 프로토콜을 직접 구현·검증하고, 정상 결제뿐 아니라 취소·도중단절·사용자취소·호스트타임아웃 등 비정상 시나리오까지 단계적으로 테스트했다. 그 과정에서 **운영에 그대로 배포했다면 "돈은 빠졌는데 취소가 안 되는" 사고를 일으켰을 결정적 버그를 발견·차단**했고, 이 발견을 코드와 회귀 테스트로 영구 고정시켰다.

핵심 결과:
- pytest **87/87 통과** (프로토콜 50 + 응답 판정 25 + NAK 재전송 12)
- 코드 라인 수 35% 감소 (1300 → 846 LOC, 3중 복붙 제거)
- 실측 응답 10종을 fixture로 잠금 → 향후 회귀 자동 검증
- 가장 큰 발견: **취소 시 S12에 넣을 승인번호는 S07이 아니라 R09**

## 2. 작업 배경

기존 레포에는 단말 정보 조회(FB/14/02)까지만 구현된 Python CLI가 있었고, 결제(FB/14/04) 흐름의 정상/비정상 응답 패턴이 미검증 상태였다. 운영 배포 전에 다음을 확보해야 했다:
1. 정상 결제·취소 동작 검증
2. 거래 도중 카드 빼기, 사용자 취소, 단말 타임아웃 등 비정상 케이스 응답 패턴 학습
3. 발견된 규칙을 코드/테스트로 잠가서 회귀 방지
4. 코드 품질 정리(중복 제거, NAK 재전송 정책)

## 4. 진행 단계

### Phase 1 — 프로토콜 단위 테스트 (HW 없이)

`ed721_proto.py` 공통 모듈을 만들고 50개 pytest로 잠금. 실측 패킷의 CRC가 모듈 출력과 byte-perfect 일치함을 확인.

- CRC16 (poly 0x8005, init 0xFFFF, LSB-first, ~ + byte-swap, 범위 LEN..ETX)
- 패킷 빌더: 16개 framing 조합(len_mode 0~3 × endian 4)
- 파서: 선행 ACK 무시, 부분 수신, CRC mismatch, ETX 손상, 다중 STX 처리
- 단말정보 79바이트 디코더 (model/version/serial/secure_id/tid/ip)

### Phase 2 — 비파괴 하드웨어 검증

단말과 연결된 상태에서 결제 외 명령들 검증(info×2, init, display, sound, RFID, 잘못된 framing→NAK).

핵심 관찰:
- **단말 자체 응답 CNT가 별도로 증가** (송신 CNT와 무관)
- **`beep.wav`만 재생** 됨, 다른 wav는 ACK만 받고 무음 (파일 부재)
- **RFID 실패 시 단말이 자동으로** "결제 취소" 화면 + "죄송합니다" 음성 출력
- 응답 없는 명령(init/display/sound)은 **ACK만으로 성공 판정** 필요

### Phase 3 — 실거래 시나리오

본인 카드로 1원/10원 승인 + 즉시 취소를 진행하면서 응답 포맷 학습. 도중 단절·사용자 취소·단말 타임아웃·중간 명령 인터럽트도 모두 실측.

#### 단말 실패 응답 3종 (모두 RCD=FF)
| DATA | 트리거 |
|------|--------|
| `CANCELED` | 빨간버튼 / 진행 중 거래 인터럽트 |
| `TIMEOUT` | 단말 자체 응답 시한 초과 |
| `FAIL` | 카드 처리 중 단절 / LA 실패 / S10 누락 |

#### 거래 인터럽트 동작
승인 진행 중 다른 명령(예: display)을 보내면 **단말이 거래 자체를 취소**시킴(단순 무시 X). POS는 한 거래가 끝나기 전 절대 다른 명령 송신 금지.

#### 시리얼 포트 다중 오픈 문제
Linux는 같은 시리얼 포트를 여러 프로세스가 동시에 열 수 있어 RX 바이트가 무작위 분배되는 사고 가능. `fcntl.flock`으로 락 필수.

#### 🚨 가장 중요한 발견: S12 = R09

10원 실거래 + 취소 시도에서 발견:
- 1차 시도: `S12 = S07 (= 949094, 단말 placeholder)` → RCD=00이지만 **R17="승인 거래 없음", R19="전표:효력없음"** → **돈은 빠진 채 취소 실패**
- 2차 시도: `S12 = R09 (= 30044993, 실 PG ID)` → R17 부재, R19="KICC로제출" → **카드 환불 확인**

이 단말은 모든 거래에 S07로 동일한 placeholder(949094)를 반환한다. 실제 PG 거래 식별자는 R09에 있고, 취소 시 반드시 R09를 S12에 넣어야 한다. **RCD=0x00만 보고 성공 판정하면 안 됨** — R17 부재 + R19 검증 필요.

#### 단말 등록 상태 변화
TID 재등록 전후로 응답 양상이 달라졌고, **현재 단말은 실 PG 매입 모드**다.

| 시그널 | 재등록 전 (실 청구 X) | 재등록 후 — 현재 (실 청구 O) |
|-------|----------------------|------------------------------|
| R09 (승인 응답) | `00000000` | 비제로 (예: 30044993) |
| R19 (승인 응답) | "매입불가...포인트리..." | "KICC로제출" |
| 실제 청구 | 없음 | 있음 (10원 출금→환불 확인) |

운영상 의미: 응답 자체로 실 매입 여부를 판단하는 것은 위험하다. **R09 비제로 + R19="KICC로제출" 조합을 실 매입 시그널로 사용하되, 단말 등록 상태가 바뀌면 양상이 달라질 수 있음을 인지**해야 한다.

#### LA(직전거래 재전송) 동작
이 단말은 **등록 상태와 무관하게 LA 미지원**:
- 재등록 전 (실 매입 X 상태): 카드 뺀 채 LA → RCD=FF, "FAIL"
- 재등록 후 (실 매입 O 상태, 10원 승인 직후): LA → RCD=FF, "FAIL"

따라서 이 단말 개체의 한계로 보임. 부수 관찰: 카드 꽂힌 채 LA 보내면 단말이 "카드 빼주세요" 안내 후 처리 시도. 다른 ED-721 단말에서도 동일 동작인지 운영 환경에서 비교 검증하면 SPEC 미지원/구현 누락 여부를 가릴 수 있음.

#### 입력 검증
단말이 즉시 거절하는 항목(즉시 FAIL): `S10 누락`, 세미콜론 구분자 누락. 그 외(`S00 누락`, `S01=ZZ`, `0원`, `99억원`, 빈 DATA)는 거래 시작됨 → **POS 측 사전 검증 필수**.

#### 잘못된 CRC
6개 손상 케이스 모두 NAK(`15 15 15`) 응답. 일관된 동작 확인.

#### 연속 송신
info를 0초 간격 10번 burst → 10개 정확히 응답, byte 손실 0. 단말 큐잉 처리 안정. 단 거래 명령은 burst 절대 금지.

#### 이미지 다운로드 (FB/14/14)
- 외부 인터넷 차단 (HTTPS·외부 HTTP 모두 RCD=FF)
- LAN 안의 HTTP 서버는 OK
- **이미지 변경은 영구**. init으로 복원 안 됨. KICC 펌웨어 기본 이미지 손실 위험
- 운영 단말에 image 명령 함부로 보내면 안 됨
- 검증 환경의 단말 홈 화면을 회색(50,50,50) 480x320 PNG로 정리해 둠

### Phase 4 — 코드 보강

#### 4-A. 응답 판정 헬퍼 + 회귀 fixture
- `parse_kv_response(data)` — EUC-KR 인식 + key=value dict 변환
- `is_approval_success(fields)` — S01=I1 + R02=A + R09 비제로
- `is_cancel_success(fields)` — S01=I4 + **R17 부재** + R09 존재 + R19 정상
- `extract_cancel_info(fields)` — 승인 응답 → `{S12: R09, S13: R07[:6], S10}` (S07 절대 X)
- `build_cancel_data(fields, pos_id)` — D4 명령 자동 생성
- `is_failure_short_message(data)` — CANCELED/TIMEOUT/FAIL 분류

회귀 테스트 25개로 잠금. 핵심 단언:
```python
assert b"S12=30044993" in cancel_data    # R09
assert b"S12=949094" not in cancel_data  # S07 절대 X
assert is_cancel_success(failed_cancel_kv) is False  # RCD=00이어도 실패
```

#### 4-B. CLI 개선
- ACK-only 명령(init/display/sound/image) 정상 처리 (이전엔 exit 2)
- 승인/취소 응답에 자동 비즈니스 판정 출력 + 취소용 S12/S13/S10 자동 안내
- `fcntl.flock` 시리얼 단일 점유 락
- `--force-retry-financial` 안전 오버라이드 플래그

#### 4-C. 리팩토링 (3중 코드 중복 제거)
| 파일 | 이전 LOC | 이후 LOC | 감소 |
|------|---------|---------|------|
| ed721_cli.py | 427 | 250 | 41%↓ |
| ed721_menu.py | 313 | 109 | 65%↓ |
| kicc_info_request.py | 360 | 125 | 65%↓ |
| **합계** | **1300** | **846** | **35%↓** |

세 파일에서 중복돼 있던 `calc_crc`, `Framing`, `ParsedPacket`, `build_packet`, `try_parse_first_packet`, `decode_terminal_info_fields`, `_request_once` 모두 제거하고 `ed721_proto`에서 import.

#### 4-D. NAK 자동 재전송 정책

설계 원칙: **중복 청구 사고 방지 최우선**.

| 명령 | 정책 | 이유 |
|------|------|------|
| FB/14/04 (승인/취소/LA) | **NEVER** | 중복 청구 위험 |
| FB/14/01·02·03·13·14 | SAFE (3회) | 무해/멱등 |
| FB/01/06 (RFID) | SAFE (3회) | 카드 SNO 재요청 무해 |

추가 안전장치:
- **순수 타임아웃은 재시도 안 함** (단말이 첫 패킷 처리 중일 수 있음)
- 단일 바이트 NAK(`0x15`)도 인식
- CLI 출력에 `attempts` 카운트 노출

운영 보호 단언:
```python
sent_packets = [t for t in ser.tx_log if t != ACK_TRIPLE]
assert len(sent_packets) == 1, "Financial command MUST never auto-retry"
```
누군가 실수로 승인 NAK→재전송 코드를 넣어도 즉시 fail.

## 5. 핵심 발견사항 (운영 반영 필수)

### 5.1 취소 시 진짜 승인번호는 R09
- S07 = 단말 placeholder (이 단말은 항상 949094 반환)
- R09 = 실 PG 거래 식별자 → **취소의 S12에 이 값 송신**
- 이 규칙이 안 지켜지면 "RCD=00 + 돈 안 돌아옴" 사고

### 5.2 취소 성공 판정 규칙
RCD=0x00 단독 신뢰 금지. 다음 모두 확인:
- RCD = 0x00
- S01 = "I4"
- **R17 키 부재** (있으면 무조건 실패)
- R19 ≠ "전표:효력없음"
- R09 채워짐
- R16 = 소문자 `d`

### 5.3 승인 성공 판정
- RCD = 0x00
- S01 = "I1", R02 = "A"
- R09 비제로 (모두 0이면 PG 매입 안 된 상태 → 단말 등록/설정 점검 필요)

### 5.4 거래 진행 중 다른 명령 송신 금지
display조차 거래를 인터럽트한다. POS는 한 거래가 끝나기 전 다른 명령 절대 X.

### 5.5 단말 응답 CNT 신뢰 금지
빨간버튼/홈 복귀 시 1로 리셋. 단말 자체 카운터로 거래 매칭하면 안 됨.

### 5.6 시리얼 포트 단일 점유 락 필수
Linux는 다중 오픈 차단 안 함. `fcntl.flock` 미적용 시 RX 바이트 분배 사고.

### 5.7 NAK 자동 재전송 — 거래 명령은 NEVER
거래 명령은 단말이 이미 처리했을 가능성을 배제 못 함. 사람 확인 후에만 수동 재전송.

### 5.8 이미지 변경은 영구
한 번 보내면 init으로 복원 불가. 운영 단말에 함부로 image 명령 X.

## 6. 운영 가이드

### 6.1 결제 처리 흐름
1. 승인 명령 전송 (FB/14/04, S01=D1, NAK 재전송 X)
2. 응답 수신 후 `is_approval_success(kv)` 검증
3. 성공이면 `extract_cancel_info(kv)` 결과를 **반드시 영속 보관** (S12=R09, S13=YYMMDD, S10=금액)
4. 취소 필요 시 `build_cancel_data(...)` 로 D4 명령 생성
5. 응답 수신 후 `is_cancel_success(kv)` 검증
6. 실패 응답 시 자동 재전송 절대 X — 사용자에게 보고

### 6.2 사전 입력 검증 (POS 측)
| 항목 | 규칙 |
|-----|------|
| S00, S01, S10, S23 | 모두 필수 |
| S01 화이트리스트 | D1, D2, D4, LA |
| S10 범위 | 1 ~ 99,999,999 |
| 모든 필드 종결자 | `;` |

### 6.3 시리얼 포트 보호
- POS 프로세스는 시작 시 `fcntl.flock` 획득
- 락 보유 못하면 즉시 종료 (다른 인스턴스 실행 중)

### 6.4 이미지 교체 (운영 단말은 권장 X)
1. POS 측에 LAN HTTP 서버 띄움 (`python3 -m http.server <port>`)
2. `image http://<POS_IP>:<port>/<file>.png` 명령
3. RCD=0x00 + 서버 액세스 로그 200 확인
4. 서버 종료

## 9. 결론

이번 작업으로 KICC ED-721 결제 모듈에 대해 **프로토콜 단위부터 실거래까지 일관된 검증 체계**를 구축했다. 가장 큰 성과는 **R09 vs S07 발견과 RCD=00의 함정**을 코드와 테스트로 영구히 잠근 것이다. 이 두 가지는 운영 환경에서 발견되었다면 카드사 분쟁/회계 사고로 이어졌을 가능성이 높았다.

남은 과제(실 운영 단말 LA 검증, 거절 응답 패턴 등)는 별도 운영 단말에서 후속 검증하면 된다. 현재 상태로 운영 통합 진행 시 **본 보고서의 §6 운영 가이드**를 그대로 따를 것을 권한다.
