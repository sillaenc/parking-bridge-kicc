# Phase 3 추가 — 잘못된 DATA / 잘못된 CRC / 연속 송신

실행: `tests/manual_phase3_extra.py`, 결과 JSON: `tests/fixtures/phase3_extra_results.json`

## A. 잘못된 DATA — 단말의 입력 검증 동작

| 케이스 | 응답 | 분류 |
|--------|------|------|
| A1: `S00` 누락 | ACK only, 본문 없음 | 거래 시작됨 (wait 부족으로 종결 못 봄) |
| A2: `S01` 누락 | RCD=FF, "CANCELED" | 거래 시작됐다가 자체 취소 |
| A3: `S10` 누락 | **RCD=FF, "FAIL"** | 즉시 거절 — S10은 단말이 형식 검증함 |
| A4: `S01=ZZ` (미정의) | ACK only | 거래 시작됨 |
| A5: 금액 0원 (`S10=0000000000`) | RCD=FF, "CANCELED" | 0원도 거래 시작은 함 |
| A6: 금액 9999999999 | ACK only | 거래 시작됨 (즉시 거절 안함) |
| A7: 빈 DATA | RCD=FF, "CANCELED" | 거래 시작 후 취소 |
| A8: 구분자 누락(`;` 없음) | **RCD=FF, "FAIL"** | 형식 오류 즉시 거절 |

### 결론
- **단말의 즉시 검증 항목** (즉시 FAIL):
  - `S10` 필수 (없으면 즉시 FAIL)
  - 세미콜론 구분자 필수 (없으면 FAIL)
- **검증 안 되는 항목** (거래 시작됨, 사용자 입력 후에야 결과 나옴):
  - `S00` 누락
  - `S01` 잘못된 값 / 누락
  - 0원 / 매우 큰 금액
  - 빈 DATA
- **POS 측 사전 검증 권장 항목**:
  - 모든 S00, S01, S10, S23 필수 체크
  - S01 값 화이트리스트 (D1/D2/D4/LA만)
  - 금액 범위 체크 (`1` ~ `99999999` 권장)
  - 세미콜론 종결 확인

## B. 잘못된 CRC — NAK 정확 동작

| 케이스 | 응답 | 비고 |
|--------|------|------|
| B1: CRC 마지막 바이트 비트 반전 | `15 15 15` | 표준 NAK |
| B2: ETX 바이트 0x00으로 손상 | `15 15 15` | 표준 NAK |
| B3: CRC 2바이트 통째 누락 | `15` (1B만) | 비표준 응답 |
| B4: STX 두 번 연속 (`02 02 ...`) | `15 15 15` | 표준 NAK |
| B5: 임의 CRC 값 | `15 15 15` | 표준 NAK |
| B6: STX 1바이트만 | `15 15 15` | 표준 NAK |

### 결론
- **유효하지 않은 프레임은 모두 NAK 응답**. 단말 동작 일관됨.
- **B3만 NAK 1바이트**: 아마 단말이 패킷 길이를 충분히 받기 전에 timeout 처리.
- POS의 NAK 재전송 정책: 단순 NAK 카운트로 판단 가능. 단 **승인/취소는 자동 재전송 금지** (중복 처리 위험 — Phase 4에서 강조).

## C. 연속 송신 — info x10 빠르게

| 항목 | 값 |
|-----|-----|
| 송신 패킷 수 | 10 |
| 파싱 성공 패킷 수 | **10** |
| 잔여 버퍼 | 0B |
| 누락 / 손실 | 없음 |

### 결론
- 단말은 **명령을 큐잉 처리**하며 byte 손실 없음
- POS 측은 응답 수신 시 ACK를 빠짐없이 보내야 함 (이번 테스트는 보냄)
- info처럼 단순한 명령은 안전하게 burst 가능
- 승인 같은 거래 명령은 burst 절대 금지 (이전 거래 진행 중에 다른 명령 보내면 거래 취소 — Phase 3-F)

## 종합 운영 가이드 (Phase 4 코드에 반영 권장)

1. **POS 측 사전 validation**:
   ```
   - S00, S01, S10, S23 필수
   - S01 ∈ {D1, D2, D4, LA}
   - S10 ∈ [1, 99999999]
   - 모든 필드 ; 종결
   ```
2. **NAK 응답**: 회사 정책에 따라 자동 재전송(info/init 등 무해 명령만) 또는 사용자 확인 후 재전송
3. **거래 명령은 직렬 처리**: 한 거래가 완전히 끝나기 전 다음 명령 송신 금지
4. **Burst 가능 명령**: info, init, display(주의) — 단 거래 진행 중에는 절대 X
