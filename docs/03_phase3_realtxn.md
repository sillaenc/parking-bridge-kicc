## 실행 시나리오

### 3-A. 사용자 취소 (카드 없이)
- TX DATA: `S00=002;S01=D1;S02=40;S09=00;S10=0000000001;S23=POS001;`
- 사용자 동작: 단말 "카드 꽂아주세요" 화면 → 빨간버튼
- RX: ACK + `02 00 0F 06 FB 14 04 FF 43 41 4E 43 45 4C 45 44 03 3C A2`
- 파싱: **RCD=FF, DATA="CANCELED"** (RFID 취소 패턴과 동일)
- device CNT = 6
- ✅ 결제 발생 없음

### 3-D-1. 1원 정상 승인
- TX DATA: `S00=002;S01=D1;S02=40;S09=00;S10=0000000001;S23=POS002;`
- 사용자 동작: 카드 삽입 + PIN 입력
- RX: 352B (ACK 3B + 응답 349B)
- **RCD=00, 응답코드 S01=I1**

### 3-D-2. 즉시 취소 (D4)
- TX DATA: `S00=002;S01=D4;S02=40;S09=00;S10=0000000001;S12=949094;S13=260429;S23=POS003;`

- RX: 397B
- **RCD=00, 응답코드 S01=I4**

## 취소에 필요한 최소 정보 (운영 핵심)
승인 응답에서 **반드시 보관**해야 할 필드:
- `S07` (= R23) → 승인번호 → 취소 시 `S12`
- `R07` 앞 6자리 → 거래일자 YYMMDD → 취소 시 `S13`
- `S10` 금액 → 취소 시 동일 금액 송신

## 한글 인코딩
- 모든 한글 응답값은 **EUC-KR**. UTF-8로 디코드하면 깨짐.
- `parsed.data.decode("euc-kr", errors="replace")` 가 맞음.
- 현재 CLI는 ASCII 디코드만 함 → Phase 4에서 EUC-KR 옵션 추가 필요.

### 시나리오: 10원 승인 → 취소 시도 (실 PG 매입 모드)

**1차 시도 — 실패** (`S12=S07=949094` 사용):
- 응답 RCD=00, S01=I4 (단말 레벨 OK)
- **그러나 R17="= IC신용반품 = 승인 거래 없음", R19="전표:효력없음"**
- 화면에도 "승인거래 없음" 표시
- **카드 통장에서 10원 빠진 채 환불 안 됨**

**2차 시도 — 성공** (`S12=R09=30044993` 사용):
- 응답 RCD=00, S01=I4
- **R17 필드 자체 없음, R19="KICC로제출"** (승인과 동일 패턴)
- R09=30044993 응답에도 채워짐, R16=`d`(소문자)
- **카드사 SMS 취소 알림 + 통장 10원 환불 확인**

### 결론: 취소 시 진짜 승인번호는 `R09`

| 필드 | 의미 | 취소 시 사용 |
|------|------|-------------|
| `S07` | 표면 승인번호 (이 단말은 fixed `949094` 반환 — placeholder) | ❌ 절대 사용 금지 |
| `R09` | **PG 거래 식별자(real approval number)** | ✅ S12에 이 값 송신 |
| `R23` | S07과 동일 값 (`949094`) | ❌ |

### 취소 성공/실패 판정 규칙 (RCD=00만 보면 안 됨!)

**성공 판정** (모두 만족):
- RCD = 0x00
- S01 = "I4"
- **R17 키 부재** (있으면 무조건 실패)
- R19 = "KICC로제출" 또는 유사 (실패 시 "전표:효력없음")
- R09 채워짐 (실패 시 부재)
- R16 = `d` (소문자, 실패 시 `D`)

**실패 판정** (하나라도 해당):
- R17 메시지 존재 (특히 "승인 거래 없음" 류)
- R19 = "전표:효력없음"
- R09 부재

## 추가 시나리오 결과 (재등록 전 — 실 매입 안 되던 시점 진행)

### 3-B. 단말 자체 타임아웃
- 카드 안 꽂고 사용자 무반응
- RX: `02 00 0E 09 FB 14 04 FF "TIMEOUT" 03 F4 32`
- **RCD=FF, DATA="TIMEOUT"** (8B ASCII)
- device CNT=9

### 3-C. 카드 인식 중 단절
- 카드 꽂은 직후 빼기
- RX: `02 00 0B 0A FB 14 04 FF "FAIL" 03 06 9A`
- **RCD=FF, DATA="FAIL"** (4B ASCII)
- device CNT=10

### 단말 실패 응답 패턴 종합
| DATA 값 | 의미 |
|---------|------|
| `CANCELED` | 사용자 빨간버튼 또는 자발적 취소 |
| `TIMEOUT` | 단말 자체 응답 시한 초과 |
| `FAIL` | 카드 처리 중 오류/단절 |

3종 모두 **RCD=FF + 짧은 ASCII 사유**. 코드는 이 3개 텍스트 분기로 처리하면 됨.

### 3-E. LA(직전거래 재전송)
- 90초 wait인 승인 한번이 단말 측에서 응답 못 보내고 빨간버튼 복귀 후
- LA 명령 송신: `S00=002;S01=LA;S02=40;S23=POS3E2;`
- RX: RCD=FF, DATA="FAIL"
- **device CNT 0x01 (=1)로 리셋됨** — 빨간버튼/홈 복귀 시 단말 응답 카운터 리셋

핵심 발견:
- LA가 항상 직전 결과를 주는 것은 아님. 단말 상태에 따라 "FAIL" 응답 가능
- 이 단말은 빨간버튼 복귀 시 응답 CNT가 1로 리셋됨

### 3-F. 거래 진행 중 다른 명령 끼워넣기 (단일 프로세스 정밀 재현)

**시나리오**: `tests/manual_3f_interrupt.py` — 같은 시리얼 핸들로 approval 송신 후 3초 뒤 display 인젝션.

**관찰**:
- 승인 명령 → 단말 화면 "카드를 대주세요" 표시 (RFID 모드)
- 3초 후 display 명령 도착
- **display 글자는 화면에 안 뜸**
- 단말은 거래를 취소하고 **홈 화면으로 복귀**
- POS 수신: 단일 패킷 RCD=FF, DATA="CANCELED"

**결론 (확정)**: POS는 **승인 진행 중에 어떤 명령도 보내면 안 됨**. display조차 거래 취소를 유발한다.

### 부수 발견 (운영 중요)

1. **Linux 시리얼 포트 다중 오픈 가능**
   - Windows와 달리 Linux는 시리얼을 두 프로세스가 동시에 열도록 허용
   - RX 바이트가 무작위 분배되어 데이터 신뢰성 깨짐
   - 1차 시도(두 프로세스)에서 본 byte 분리 현상이 이 때문
   - 운영: `fcntl.flock` 단일 점유 락 필수 (Phase 4 작업)

2. **카드 꽂힌 채 LA 보내면 "카드 빼라" 화면 먼저**
   - 사용자가 카드 빼야 LA 처리 진행
   - LA wait는 30초+ 권장

### 3-E 재시도. 완전 성공 승인 + LA 회수
- `tests/manual_3e_la_retrieve.py` — 단일 시리얼 핸들로 승인 → LA 시퀀스
- 1차 LA 송신: 카드 꽂힌 채 → 단말 "카드 빼주세요" → 우리 wait 15초 끊김
- 2차 LA 송신: 카드 뺀 후 → **RCD=FF, DATA="FAIL"**
- **결론**: 이 테스트 단말은 LA 미지원 (또는 거래 이력 영구 보관 안 함). 운영 단말에서 재검증 필요.
